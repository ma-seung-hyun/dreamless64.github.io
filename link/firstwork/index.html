시멘틱태그를안쓴다

구획은 일단 DIV.

어째서인지 header나 nav같은 시멘틱 태그로 문서를 구성하지 않은 모습이 1차적으로 눈에 띄었습니다.
그래도 id와 class의 네이밍으로 이런 구조를 모두 세세하게 구분하고 있었습니다.
헤더 이후의 메인 콘텐츠들은 section > article > div 순으로 뎁스 관계를 설정해 두었으며
h1의 설정을 로고가 아닌 문서 전체의 가장 상위뎁스 항목에 두었습니다.

  1. 따로 사용하지 않는 reset css. 역시 커스텀이 중요.
  2. 모든 문서를 .warp으로 감싸고 시작한다. flex를 사용?
  3. 네비게이션 영역이 window의 전체 사이즈? 이유는 무엇일까
해당 페이지
  • INLINE
    스타일을 작성할때, 셀렉터와 내용은 줄바꿈을 하지만 속성과 값은 인라인형태로의 작성을 하는 모습입니다.
  • STYLE SCOPED
    네비게이션등을 제외한 메인 컨텐츠에 해당하는 부분에서의 스타일은 scoped속성이 적용된 style태그로 구분해서 사용했습니다.
  • POSITION
    position속성을 제 기준보다 많이 사용하는 느낌입니다. 개인적으로는 서로가 맞닿아있는 웹의 형태가 가장 이상적이라고 생각해 사용은 하되 최대한 지양하고 있습니다.
  • MEDIA QUERY
    완벽한 적응형이 아닌 반응형 이미지 변경 요소들이 있지만 media query를 사용하진 않은걸보아 스크립트나 다른 방식으로 동작하게 한 모양입니다.
  • IMG & FIGURE
    img 태그를 그냥 쓰는법이 없습니다. 대부분 figure 태그로 한번 감싸주고, figcaption 태그를 사용하지 않는 경우에도 어김없이 한번 감싸주는 형태입니다.
  • ! IMPORTENT
    스크립트로 엘리먼트에 직접 스타일을 입력하는것 이상으로 우선순위를 매길만한 상황이 이렇게 많았는지 몰랐습니다. 약 200여개의 !importent가 사용되었습니다.

모르는부분, 공부가 필요하다.

프로젝트 성격상 모두 '정보형' 이미지라고 판단했는지 background-image 대신 전부 img태그를 사용했습니다.
연동 마크업
스크립트 애니메이션 효과를 염두에두고 해당 애니메이션에 가장 적합한 형태로의 마크업을 진행했습니다.
알수없는 table
모든 문서 하단에 알수없는 table태그가 있습니다. tbod,tr,td등 필요한 태그들도 채워진것을 보니 어떤 용도가 있는것이 분명해보입니다.
